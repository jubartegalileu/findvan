# Story 11.2 — Messaging Receipts/Webhooks Contract

**Epic:** 11 (Wave 7 Campaign Cadence Operations)
**Story ID:** 11.2
**Status:** Ready
**Priority:** P1
**Complexity:** Média (5 pontos)

## Summary

Definir e implementar contrato incremental para receipts/webhooks de entrega e resposta da mensageria, sem quebra de compatibilidade com os fluxos atuais.

## Acceptance Criteria

1. Contrato incremental de receipts/webhooks
- [ ] Contrato documentado para eventos de `delivered`, `failed`, `replied` com campos mínimos obrigatórios.
- [ ] Endpoint de recebimento aceita payload versionado e valida dados essenciais.
- [ ] Eventos inválidos retornam erro claro sem degradar endpoints existentes.

2. Backward compatibility
- [ ] Endpoints existentes das Waves 1-6 permanecem compatíveis.
- [ ] Ausência de webhook não bloqueia envio já funcional em WhatsApp/Leads.
- [ ] Estratégia de fallback documentada para ambientes sem callback de provider.

3. Qualidade e rastreabilidade
- [ ] Testes de contrato cobrindo payload válido/inválido.
- [ ] Evidência de gate arquitetural aprovando sem breaking change.

## API Contract (PO detail)

- Novo endpoint de callback (a confirmar na implementação) deve:
  - aceitar versão de contrato explícita;
  - registrar receipt com referência de mensagem/lead/campanha;
  - preservar semântica das APIs já em uso.

## Tasks

- [ ] Definir schema de receipt/webhook e versionamento incremental
- [ ] Implementar endpoint + validação mínima de payload
- [ ] Persistir eventos para consumo operacional
- [ ] Criar testes de contrato/compatibilidade
- [ ] Submeter gate @architect e validação @qa

## Definition of Done

- [ ] Contrato de receipts/webhooks publicado e implementado sem breaking change
- [ ] Gate @architect aprovado
- [ ] QA valida fluxo compatível sem regressão crítica
- [ ] `npm run lint`, `npm run typecheck`, `npm test` verdes

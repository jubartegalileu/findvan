# Story 11.2 — Messaging Receipts/Webhooks Contract

**Epic:** 11 (Wave 7 Campaign Cadence Operations)
**Story ID:** 11.2
**Status:** In Review
**Priority:** P1
**Complexity:** Média (5 pontos)

## Summary

Definir e implementar contrato incremental para receipts/webhooks de entrega e resposta da mensageria, sem quebra de compatibilidade com os fluxos atuais.

## Acceptance Criteria

1. Contrato incremental de receipts/webhooks
- [x] Contrato documentado para eventos de `delivered`, `failed`, `replied` com campos mínimos obrigatórios.
- [x] Endpoint de recebimento aceita payload versionado e valida dados essenciais.
- [x] Eventos inválidos retornam erro claro sem degradar endpoints existentes.

2. Backward compatibility
- [x] Endpoints existentes das Waves 1-6 permanecem compatíveis.
- [x] Ausência de webhook não bloqueia envio já funcional em WhatsApp/Leads.
- [x] Estratégia de fallback documentada para ambientes sem callback de provider.

3. Qualidade e rastreabilidade
- [x] Testes de contrato cobrindo payload válido/inválido.
- [x] Evidência de gate arquitetural aprovando sem breaking change.

## API Contract (PO detail)

- `POST /api/integrations/messaging/receipts`
  - aceita versão de contrato explícita (`version`) e valida aliases suportados;
  - registra evento de receipt com referência de mensagem/lead/campanha quando fornecido;
  - mantém semântica das APIs já em uso, sem alterar fluxo de envio.
- `GET /api/integrations/messaging/receipts`
  - expõe eventos mais recentes para consumo operacional.

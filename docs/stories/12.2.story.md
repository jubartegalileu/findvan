# Story 12.2 — Webhook Idempotency Policy

**Epic:** 12 (Wave 8 Receipts Persistence & Reconciliation)
**Story ID:** 12.2
**Status:** Done
**Priority:** P1
**Complexity:** Média (5 pontos)

## Summary

Definir e implementar política de idempotência para callbacks de webhook, evitando processamento duplicado por retries de provider e preservando compatibilidade de contrato.

## Acceptance Criteria

1. Idempotência
- [x] Eventos repetidos com mesma chave idempotente não duplicam processamento/contagem.
- [x] Política explícita para `external_id` + provider + tipo de evento aplicada no backend.
- [x] Resposta de webhook permanece consistente para reentregas.

2. Compatibilidade
- [x] Contrato atual de receipts/webhooks permanece compatível.
- [x] Fluxo de monitoramento não sofre regressão com deduplicação ativa.

3. Qualidade e evidência
- [x] Testes cobrindo evento novo, duplicado e variação legítima de evento.
- [x] Gate arquitetural aprova estratégia sem breaking change.

## API Contract (PO detail)

- `POST /api/integrations/messaging/receipts`
  - mantém payload atual;
  - aplica idempotência por chave `external_id + provider + event_type`;
  - não quebra semântica de respostas já consumidas.
  - inclui `idempotent` (`true/false`) para sinalizar reentrega deduplicada.

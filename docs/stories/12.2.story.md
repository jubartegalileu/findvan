# Story 12.2 — Webhook Idempotency Policy

**Epic:** 12 (Wave 8 Receipts Persistence & Reconciliation)
**Story ID:** 12.2
**Status:** Ready
**Priority:** P1
**Complexity:** Média (5 pontos)

## Summary

Definir e implementar política de idempotência para callbacks de webhook, evitando processamento duplicado por retries de provider e preservando compatibilidade de contrato.

## Acceptance Criteria

1. Idempotência
- [ ] Eventos repetidos com mesma chave idempotente não duplicam processamento/contagem.
- [ ] Política explícita para `external_id` + provider + tipo de evento aplicada no backend.
- [ ] Resposta de webhook permanece consistente para reentregas.

2. Compatibilidade
- [ ] Contrato atual de receipts/webhooks permanece compatível.
- [ ] Fluxo de monitoramento não sofre regressão com deduplicação ativa.

3. Qualidade e evidência
- [ ] Testes cobrindo evento novo, duplicado e variação legítima de evento.
- [ ] Gate arquitetural aprova estratégia sem breaking change.

## API Contract (PO detail)

- `POST /api/integrations/messaging/receipts`
  - mantém payload atual;
  - aplica idempotência por chave definida;
  - não quebra semântica de respostas já consumidas.

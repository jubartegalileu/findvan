# Story 13.2 — Receipts Retention & Query Limits

**Epic:** 13 (Wave 9 Scale, Observability & Cost Control)
**Story ID:** 13.2
**Status:** Done
**Priority:** P1
**Complexity:** Média (5 pontos)

## Summary

Implementar política de retenção para receipts/eventos e limites de consulta para estabilizar performance de leitura e evitar degradação operacional com crescimento de volume.

## Acceptance Criteria

1. Política de retenção
- [x] Política de retenção definida e aplicada para receipts/eventos com janela explícita.
- [x] Processo de limpeza não remove dados fora da política definida.
- [x] Operação principal continua funcional após ciclos de retenção.

2. Limites de consulta
- [x] Endpoints de leitura críticos aplicam limites/paginação defensiva.
- [x] Consultas fora do limite retornam resposta previsível e compatível.
- [x] Não há breaking change para consumidores existentes.

3. Estabilidade e compatibilidade
- [x] Evidência de melhora/estabilidade de leitura em cenários de maior volume.
- [x] Contratos atuais preservados (campos e semântica).
- [x] Gate arquitetural aprova solução com foco em escala.

## API Contract (PO detail)

- Endpoints existentes devem manter formato de resposta.
- Limites default e máximos devem ser documentados e backward compatible.
- Não alterar semântica de filtros existentes; apenas restringir excessos.

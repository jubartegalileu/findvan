# Story 13.2 — Receipts Retention & Query Limits

**Epic:** 13 (Wave 9 Scale, Observability & Cost Control)
**Story ID:** 13.2
**Status:** Draft
**Priority:** P1
**Complexity:** Média (5 pontos)

## Summary

Implementar política de retenção para receipts/eventos e limites de consulta para estabilizar performance de leitura e evitar degradação operacional com crescimento de volume.

## Acceptance Criteria

1. Política de retenção
- [ ] Política de retenção definida e aplicada para receipts/eventos com janela explícita.
- [ ] Processo de limpeza não remove dados fora da política definida.
- [ ] Operação principal continua funcional após ciclos de retenção.

2. Limites de consulta
- [ ] Endpoints de leitura críticos aplicam limites/paginação defensiva.
- [ ] Consultas fora do limite retornam resposta previsível e compatível.
- [ ] Não há breaking change para consumidores existentes.

3. Estabilidade e compatibilidade
- [ ] Evidência de melhora/estabilidade de leitura em cenários de maior volume.
- [ ] Contratos atuais preservados (campos e semântica).
- [ ] Gate arquitetural aprova solução com foco em escala.

## API Contract (PO detail)

- Endpoints existentes devem manter formato de resposta.
- Limites default e máximos devem ser documentados e backward compatible.
- Não alterar semântica de filtros existentes; apenas restringir excessos.

## Tasks

- [ ] Definir janela de retenção para receipts e activity
- [ ] Implementar rotina de limpeza/expurgo segura
- [ ] Aplicar limites default/máximos nos endpoints críticos
- [ ] Cobrir comportamento com testes de contrato e performance básica
- [ ] Executar gate @architect e validação @qa

## Definition of Done

- [ ] Política de retenção implementada e documentada
- [ ] Limites/paginação aplicados sem regressão de contrato
- [ ] Gate @architect aprovado
- [ ] QA valida regressão funcional

# Story 14.1 — Async Retention Jobs

**Epic:** 14 (Wave 10 Production Hardening, Alerting & Metrics Governance)
**Story ID:** 14.1
**Status:** Done
**Priority:** P1
**Complexity:** Média (5 pontos)

## Summary

Remover expurgo de receipts/activity do request path e implementar execução assíncrona periódica com visibilidade operacional de última execução.

## Acceptance Criteria

1. Retenção assíncrona
- [x] Expurgo de receipts e activity ocorre via job agendado (scheduler/worker) fora de requisições HTTP.
- [x] Endpoints de leitura deixam de executar expurgo a cada chamada.
- [x] Execução do job respeita janelas de retenção configuráveis.

2. Confiabilidade operacional
- [x] Job registra resultado de execução (sucesso/falha, contagem de registros removidos, timestamp).
- [x] Falhas de job não quebram fluxo principal da API.
- [x] Existe endpoint/indicador de health para última execução do job.

3. Compatibilidade e performance
- [x] Contratos atuais de endpoints permanecem compatíveis.
- [x] Latência de endpoints críticos melhora ou permanece estável após remoção de expurgo em request path.
- [x] Gate arquitetural aprova desenho de execução assíncrona.

## API Contract (PO detail)

- Endpoints existentes devem manter payloads já consumidos.
- Metadados de job podem ser expostos em endpoint adicional sem quebrar contratos atuais.
- Configuração de retenção deve permanecer backward compatible.

## Tasks

- [x] Extrair expurgo para job assíncrono/scheduler
- [x] Remover expurgo inline dos endpoints de leitura
- [x] Persistir/registrar estado de última execução do job
- [x] Expor estado operacional do job para monitoramento
- [x] Validar regressão @qa

## Definition of Done

- [x] Expurgo assíncrono ativo e estável
- [x] Endpoints críticos sem expurgo no request path
- [x] Gate @architect aprovado
- [x] QA valida regressão funcional
- [x] `npm run lint`, `npm run typecheck`, `npm test` verdes

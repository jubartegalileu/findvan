# Story 14.1 — Async Retention Jobs

**Epic:** 14 (Wave 10 Production Hardening, Alerting & Metrics Governance)
**Story ID:** 14.1
**Status:** Draft
**Priority:** P1
**Complexity:** Média (5 pontos)

## Summary

Remover expurgo de receipts/activity do request path e implementar execução assíncrona periódica com visibilidade operacional de última execução.

## Acceptance Criteria

1. Retenção assíncrona
- [ ] Expurgo de receipts e activity ocorre via job agendado (scheduler/worker) fora de requisições HTTP.
- [ ] Endpoints de leitura deixam de executar expurgo a cada chamada.
- [ ] Execução do job respeita janelas de retenção configuráveis.

2. Confiabilidade operacional
- [ ] Job registra resultado de execução (sucesso/falha, contagem de registros removidos, timestamp).
- [ ] Falhas de job não quebram fluxo principal da API.
- [ ] Existe endpoint/indicador de health para última execução do job.

3. Compatibilidade e performance
- [ ] Contratos atuais de endpoints permanecem compatíveis.
- [ ] Latência de endpoints críticos melhora ou permanece estável após remoção de expurgo em request path.
- [ ] Gate arquitetural aprova desenho de execução assíncrona.

## API Contract (PO detail)

- Endpoints existentes devem manter payloads já consumidos.
- Metadados de job podem ser expostos em endpoint adicional sem quebrar contratos atuais.
- Configuração de retenção deve permanecer backward compatible.

## Tasks

- [ ] Extrair expurgo para job assíncrono/scheduler
- [ ] Remover expurgo inline dos endpoints de leitura
- [ ] Persistir/registrar estado de última execução do job
- [ ] Expor estado operacional do job para monitoramento
- [ ] Validar com gate @architect e regressão @qa

## Definition of Done

- [ ] Expurgo assíncrono ativo e estável
- [ ] Endpoints críticos sem expurgo no request path
- [ ] Gate @architect aprovado
- [ ] QA valida regressão funcional
- [ ] `npm run lint`, `npm run typecheck`, `npm test` verdes

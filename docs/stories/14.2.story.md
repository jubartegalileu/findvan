# Story 14.2 — External Operational Alerting

**Epic:** 14 (Wave 10 Production Hardening, Alerting & Metrics Governance)
**Story ID:** 14.2
**Status:** Draft
**Priority:** P1
**Complexity:** Média (5 pontos)

## Summary

Emitir alertas críticos de SLO para endpoint externo (webhook/Slack-like), mantendo fallback local e controle de ruído operacional.

## Acceptance Criteria

1. Emissão de alertas externos
- [ ] Eventos críticos de SLO geram payload de alerta padronizado para endpoint externo configurado.
- [ ] Payload inclui severidade, métrica afetada, janela temporal e timestamp.
- [ ] Alertas são enviados de forma não bloqueante para o fluxo principal.

2. Fallback e robustez
- [ ] Se endpoint externo estiver indisponível, fallback local mantém registro do alerta.
- [ ] Falha no envio externo não causa erro funcional nas telas/APIs.
- [ ] Retry/supressão básica evita spam de alertas repetidos no curto prazo.

3. Compatibilidade e governança inicial
- [ ] Configuração de alerting é opcional e backward compatible.
- [ ] Contratos existentes permanecem estáveis sem breaking change.
- [ ] Gate arquitetural aprova contrato e política de fallback.

## API Contract (PO detail)

- Não alterar contratos existentes de monitoramento.
- Expor endpoint/config opcional para status de alerting (quando necessário).
- Payload externo deve ser versionável e documentado.

## Tasks

- [ ] Definir contrato de payload para alerting externo
- [ ] Implementar serviço de envio com timeout/retry controlado
- [ ] Implementar fallback local e supressão básica de ruído
- [ ] Expor estado operacional mínimo de alerting
- [ ] Validar com gate @architect e regressão @qa

## Definition of Done

- [ ] Alertas externos críticos funcionais com fallback local
- [ ] Ruído de alertas controlado por supressão/retry
- [ ] Gate @architect aprovado
- [ ] QA valida regressão funcional
- [ ] `npm run lint`, `npm run typecheck`, `npm test` verdes

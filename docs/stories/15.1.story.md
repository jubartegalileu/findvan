# Story 15.1 — Dedicated Worker & Distributed Lock for Retention

**Epic:** 15 (Wave 11 Multi-Instance Readiness & Distributed Resilience)
**Story ID:** 15.1
**Status:** Done
**Priority:** P1
**Complexity:** Grande (8 pontos)

## Summary

Mover a execução de retenção para worker dedicado com lock distribuído e heartbeat, evitando corrida entre instâncias e separando job do processo API.

## Acceptance Criteria

1. Worker dedicado
- [x] Job de retenção executa em processo/worker separado do FastAPI.
- [x] API deixa de iniciar ciclo periódico de retenção internamente.
- [x] Worker expõe status operacional mínimo (running, last_run, last_success, owner).

2. Lock distribuído
- [x] Execução protegida por lock distribuído com TTL e renovação por heartbeat.
- [x] Em cenário multi-instância, apenas um owner executa o ciclo por janela.
- [x] Perda de owner permite failover seguro para outra instância.

3. Compatibilidade e confiabilidade
- [x] Endpoints existentes mantêm contrato/backward compatibility.
- [x] Falha de lock/worker não derruba API principal.
- [x] Gate arquitetural aprova desenho de isolamento e lock.

## API Contract (PO detail)

- Endpoints atuais devem preservar payload.
- Estado de worker/lock pode ser exposto por endpoint aditivo.
- Sem obrigatoriedade de novo campo para consumidores antigos.

## Tasks

- [x] Extrair runtime do retention job para worker dedicado
- [x] Implementar lock distribuído com TTL/heartbeat
- [x] Implementar status de owner e failover
- [x] Ajustar startup da API para não iniciar loop local de retenção
- [x] Validar com gate @architect e regressão @qa

# Story 15.1 — Dedicated Worker & Distributed Lock for Retention

**Epic:** 15 (Wave 11 Multi-Instance Readiness & Distributed Resilience)
**Story ID:** 15.1
**Status:** Draft
**Priority:** P1
**Complexity:** Grande (8 pontos)

## Summary

Mover a execução de retenção para worker dedicado com lock distribuído e heartbeat, evitando corrida entre instâncias e separando job do processo API.

## Acceptance Criteria

1. Worker dedicado
- [ ] Job de retenção executa em processo/worker separado do FastAPI.
- [ ] API deixa de iniciar ciclo periódico de retenção internamente.
- [ ] Worker expõe status operacional mínimo (running, last_run, last_success, owner).

2. Lock distribuído
- [ ] Execução protegida por lock distribuído com TTL e renovação por heartbeat.
- [ ] Em cenário multi-instância, apenas um owner executa o ciclo por janela.
- [ ] Perda de owner permite failover seguro para outra instância.

3. Compatibilidade e confiabilidade
- [ ] Endpoints existentes mantêm contrato/backward compatibility.
- [ ] Falha de lock/worker não derruba API principal.
- [ ] Gate arquitetural aprova desenho de isolamento e lock.

## API Contract (PO detail)

- Endpoints atuais devem preservar payload.
- Estado de worker/lock pode ser exposto por endpoint aditivo.
- Sem obrigatoriedade de novo campo para consumidores antigos.

## Tasks

- [ ] Extrair runtime do retention job para worker dedicado
- [ ] Implementar lock distribuído com TTL/heartbeat
- [ ] Implementar status de owner e failover
- [ ] Ajustar startup da API para não iniciar loop local de retenção
- [ ] Validar com gate @architect e regressão @qa

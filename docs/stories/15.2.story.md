# Story 15.2 — Persistent Operational State Store

**Epic:** 15 (Wave 11 Multi-Instance Readiness & Distributed Resilience)
**Story ID:** 15.2
**Status:** In Review
**Priority:** P1
**Complexity:** Grande (8 pontos)

## Summary

Persistir estado operacional de retenção/alerting em storage compartilhado, substituindo dependência de estado em memória para leitura consistente entre instâncias.

## Acceptance Criteria

1. Persistência de estado operacional
- [x] Estado de retenção (run/fail/last_success/last_error/owner) é persistido em banco.
- [x] Estado de alerting (sent/fallback/suppressed/recent) é persistido em banco.
- [x] API de status lê dados persistentes com fallback controlado.

2. Migração e compatibilidade
- [x] Migração de estado em memória para persistente sem quebra de contrato.
- [x] Endpoints existentes preservam formato de resposta esperado.
- [x] Rollback/fallback seguro disponível para incidentes de migração.

3. Consistência multi-instância
- [x] Leitura de estado operacional é consistente entre instâncias.
- [x] Escritas concorrentes preservam integridade do estado.
- [x] Gate arquitetural aprova desenho de schema e consistência.

## API Contract (PO detail)

- Manter payload atual de status operacional.
- Campos novos apenas aditivos quando necessário.
- Não remover chaves já consumidas pela UI.

## Tasks

- [x] Criar schema/tabelas para estado operacional de retenção/alerting
- [x] Migrar serviços para leitura/escrita persistente
- [x] Implementar fallback de migração controlado
- [x] Cobrir consistência com testes de concorrência básica
- [ ] Validar com gate @architect e regressão @qa

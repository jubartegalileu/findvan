# Story 15.2 — Persistent Operational State Store

**Epic:** 15 (Wave 11 Multi-Instance Readiness & Distributed Resilience)
**Story ID:** 15.2
**Status:** Draft
**Priority:** P1
**Complexity:** Grande (8 pontos)

## Summary

Persistir estado operacional de retenção/alerting em storage compartilhado, substituindo dependência de estado em memória para leitura consistente entre instâncias.

## Acceptance Criteria

1. Persistência de estado operacional
- [ ] Estado de retenção (run/fail/last_success/last_error/owner) é persistido em banco.
- [ ] Estado de alerting (sent/fallback/suppressed/recent) é persistido em banco.
- [ ] API de status lê dados persistentes com fallback controlado.

2. Migração e compatibilidade
- [ ] Migração de estado em memória para persistente sem quebra de contrato.
- [ ] Endpoints existentes preservam formato de resposta esperado.
- [ ] Rollback/fallback seguro disponível para incidentes de migração.

3. Consistência multi-instância
- [ ] Leitura de estado operacional é consistente entre instâncias.
- [ ] Escritas concorrentes preservam integridade do estado.
- [ ] Gate arquitetural aprova desenho de schema e consistência.

## API Contract (PO detail)

- Manter payload atual de status operacional.
- Campos novos apenas aditivos quando necessário.
- Não remover chaves já consumidas pela UI.

## Tasks

- [ ] Criar schema/tabelas para estado operacional de retenção/alerting
- [ ] Migrar serviços para leitura/escrita persistente
- [ ] Implementar fallback de migração controlado
- [ ] Cobrir consistência com testes de concorrência básica
- [ ] Validar com gate @architect e regressão @qa

## Definition of Done

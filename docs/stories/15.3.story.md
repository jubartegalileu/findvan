# Story 15.3 — Durable Governance Audit & Failover Telemetry

**Epic:** 15 (Wave 11 Multi-Instance Readiness & Distributed Resilience)
**Story ID:** 15.3
**Status:** Draft
**Priority:** P1
**Complexity:** Média (5 pontos)

## Summary

Persistir auditoria de governança e telemetria de failover (retenção/alerting), fornecendo visibilidade operacional para investigação pós-incidente.

## Acceptance Criteria

1. Auditoria durável
- [ ] Alterações de thresholds são persistidas em storage durável (não apenas memória).
- [ ] Histórico de auditoria mantém autor, timestamp, diffs e origem da alteração.
- [ ] Consulta de auditoria suporta paginação/limite previsível.

2. Telemetria de failover
- [ ] Eventos de failover/erro de retenção e alerting são registrados em storage persistente.
- [ ] Operação consegue visualizar últimos incidentes com contexto suficiente.
- [ ] Métricas de saúde operacional (falhas, suppressions, fallback rate) disponíveis para leitura.

3. Aderência de produto e regressão
- [ ] Leitura operacional está clara para triagem e pós-mortem.
- [ ] Fluxos atuais de dashboard/monitoramento permanecem sem regressão.
- [ ] Gate @po aprova aderência ao objetivo de observabilidade operacional.

## API Contract (PO detail)

- Endpoints de governança/alerting devem manter compatibilidade.
- Novos endpoints para telemetria podem ser adicionados de forma aditiva.
- Não remover campos atualmente usados pela UI.

## Tasks

- [ ] Persistir audit trail de governança em banco
- [ ] Persistir eventos de failover/supressão/fallback de alerting
- [ ] Expor endpoint(s) de telemetria operacional com limite/paginação
- [ ] Ajustar UI operacional para exibir últimos incidentes relevantes
- [ ] Submeter para gate @po e regressão @qa

## Definition of Done

- [ ] Auditoria e telemetria de failover persistidas e consultáveis
- [ ] Gate @po aprovado
- [ ] QA valida regressão funcional/visual
- [ ] `npm run lint`, `npm run typecheck`, `npm test` verdes

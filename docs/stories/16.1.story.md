# Story 16.1 — Automated Recovery Policies for Retention/Alerting

**Epic:** 16 (Wave 12 Operational Excellence & Self-Healing)
**Story ID:** 16.1
**Status:** Draft
**Priority:** P1
**Complexity:** Grande (8 pontos)

## Summary

Implementar políticas automáticas de recuperação para falhas recorrentes de retenção e alerting, com limites de segurança e rastreabilidade operacional.

## Acceptance Criteria

1. Políticas de recuperação
- [ ] Falhas recorrentes de retenção disparam política automática de recuperação controlada.
- [ ] Falhas de entrega de alerting aplicam retry/backoff com limite e cooldown.
- [ ] Circuit-breaker operacional evita loops de remediação agressiva.

2. Segurança e governança
- [ ] Cada ação de remediação é registrada com causa, ação executada e resultado.
- [ ] Execuções automáticas respeitam limites por janela (budget de recuperação).
- [ ] Possível desativar remediação automática por configuração sem quebrar fluxo principal.

3. Compatibilidade e confiabilidade
- [ ] Endpoints atuais permanecem backward compatible.
- [ ] Falha no fluxo de remediação não derruba API nem worker principal.
- [ ] Gate arquitetural aprova desenho de segurança da automação.

## API Contract (PO detail)

- Payloads atuais devem permanecer estáveis.
- Dados de remediação podem ser expostos em endpoint aditivo de telemetria.
- Sem remoção de campos atualmente consumidos pela UI.

## Tasks

- [ ] Definir políticas de auto-recuperação por tipo de incidente
- [ ] Implementar retry/backoff e circuit-breaker para remediações automáticas
- [ ] Persistir trilha de execução de remediação
- [ ] Expor status operacional de políticas de recuperação
- [ ] Validar com gate @architect e regressão @qa

## Definition of Done

- [ ] Políticas de recuperação automática ativas para cenários críticos definidos
- [ ] Limites de segurança (cooldown/budget/circuit-breaker) funcionais
- [ ] Gate @architect aprovado
- [ ] QA valida regressão funcional
- [ ] `npm run lint`, `npm run typecheck`, `npm test` verdes

# Story 16.1 — Automated Recovery Policies for Retention/Alerting

**Epic:** 16 (Wave 12 Operational Excellence & Self-Healing)
**Story ID:** 16.1
**Status:** In Review
**Priority:** P1
**Complexity:** Grande (8 pontos)

## Summary

Implementar políticas automáticas de recuperação para falhas recorrentes de retenção e alerting, com limites de segurança e rastreabilidade operacional.

## Acceptance Criteria

1. Políticas de recuperação
- [x] Falhas recorrentes de retenção disparam política automática de recuperação controlada.
- [x] Falhas de entrega de alerting aplicam retry/backoff com limite e cooldown.
- [x] Circuit-breaker operacional evita loops de remediação agressiva.

2. Segurança e governança
- [x] Cada ação de remediação é registrada com causa, ação executada e resultado.
- [x] Execuções automáticas respeitam limites por janela (budget de recuperação).
- [x] Possível desativar remediação automática por configuração sem quebrar fluxo principal.

3. Compatibilidade e confiabilidade
- [x] Endpoints atuais permanecem backward compatible.
- [x] Falha no fluxo de remediação não derruba API nem worker principal.
- [x] Gate arquitetural aprova desenho de segurança da automação.

## API Contract (PO detail)

- Payloads atuais devem permanecer estáveis.
- Dados de remediação podem ser expostos em endpoint aditivo de telemetria.
- Sem remoção de campos atualmente consumidos pela UI.

## Tasks

- [x] Definir políticas de auto-recuperação por tipo de incidente
- [x] Implementar retry/backoff e circuit-breaker para remediações automáticas
- [x] Persistir trilha de execução de remediação
- [x] Expor status operacional de políticas de recuperação
- [ ] Validar com gate @architect e regressão @qa

## Definition of Done

- [x] Políticas de recuperação automática ativas para cenários críticos definidos
- [x] Limites de segurança (cooldown/budget/circuit-breaker) funcionais
- [x] Gate @architect aprovado
- [x] QA valida regressão funcional
- [x] `npm run lint`, `npm run typecheck`, `npm test` verdes

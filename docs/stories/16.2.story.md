# Story 16.2 — SLO Degradation Guardrails & Incident Classification

**Epic:** 16 (Wave 12 Operational Excellence & Self-Healing)
**Story ID:** 16.2
**Status:** In Review
**Priority:** P1
**Complexity:** Média (5 pontos)

## Summary

Classificar degradações de SLO por severidade e impacto, expondo guardrails acionáveis para operação no backend e dashboard.

## Acceptance Criteria

1. Classificação de incidentes
- [x] Incidentes são classificados com severidade padronizada (`low`, `medium`, `high`, `critical`).
- [x] Classificação inclui impacto operacional e componente afetado (retention/alerting/messaging).
- [x] Regras de classificação são versionadas e auditáveis.

2. Guardrails operacionais
- [x] Guardrails de degradação expõem recomendações acionáveis para operação.
- [x] API retorna status de guardrail com limite/paginação previsíveis.
- [x] Dashboard exibe guardrails prioritários sem regressão das seções atuais.

3. Compatibilidade e consistência
- [x] Endpoints existentes permanecem estáveis; novos campos são aditivos.
- [x] Sinais de severidade são consistentes entre API e UI.
- [x] Gate arquitetural aprova contrato de classificação e guardrails.

## API Contract (PO detail)

- Não remover campos atuais usados pelo dashboard.
- Novos campos de severidade/guardrails devem ser aditivos.
- Contrato de telemetria deve manter limites claros (`limit`, `offset`, `applied_limit`).

# Story 16.2 — SLO Degradation Guardrails & Incident Classification

**Epic:** 16 (Wave 12 Operational Excellence & Self-Healing)
**Story ID:** 16.2
**Status:** Draft
**Priority:** P1
**Complexity:** Média (5 pontos)

## Summary

Classificar degradações de SLO por severidade e impacto, expondo guardrails acionáveis para operação no backend e dashboard.

## Acceptance Criteria

1. Classificação de incidentes
- [ ] Incidentes são classificados com severidade padronizada (`low`, `medium`, `high`, `critical`).
- [ ] Classificação inclui impacto operacional e componente afetado (retention/alerting/messaging).
- [ ] Regras de classificação são versionadas e auditáveis.

2. Guardrails operacionais
- [ ] Guardrails de degradação expõem recomendações acionáveis para operação.
- [ ] API retorna status de guardrail com limite/paginação previsíveis.
- [ ] Dashboard exibe guardrails prioritários sem regressão das seções atuais.

3. Compatibilidade e consistência
- [ ] Endpoints existentes permanecem estáveis; novos campos são aditivos.
- [ ] Sinais de severidade são consistentes entre API e UI.
- [ ] Gate arquitetural aprova contrato de classificação e guardrails.

## API Contract (PO detail)

- Não remover campos atuais usados pelo dashboard.
- Novos campos de severidade/guardrails devem ser aditivos.
- Contrato de telemetria deve manter limites claros (`limit`, `offset`, `applied_limit`).

## Tasks

- [ ] Integrar visualização de guardrails no dashboard operacional
- [ ] Cobrir regras de classificação com testes automatizados
- [ ] Validar com gate @architect e regressão @qa

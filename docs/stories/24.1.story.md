# Story 24.1 — Backend Foundation: API de Leads + Tabelas SDR/Pipeline

**Epic:** 24 (SDR & Funil de Vendas: Workflow Linear)
**Story ID:** 24.1
**Status:** Done
**Priority:** P0
**Owner:** @data-engineer
**Estimate:** TBD
**Complexity:** Média-Alta (8 pontos)

## Story

**As a** FindVan developer,
**I want** a complete backend with leads API, SDR activities table, and pipeline table with auto-creation trigger,
**so that** all lead, SDR, and funnel data is persisted in the database instead of only existing in frontend memory.

## Acceptance Criteria

1. Migration SQL
- [ ] Migration `004-create-sdr-pipeline.sql` criada e executável sem erros
- [ ] Função genérica `update_timestamp()` criada e reutilizável
- [ ] Tabela `sdr_activities` criada com campos: `id`, `lead_id` (BIGINT), `assigned_to`, `prospect_status`, `next_action_date`, `next_action_description`, `cadence_step`, `last_contact_at`, `contact_count`, `notes` (JSONB)
- [ ] Tabela `pipeline` criada com campos: `id`, `lead_id` (BIGINT), `funnel_status` (NOT NULL), `entered_stage_at`, `loss_reason`, `loss_reason_detail`, `stage_history` (JSONB)
- [ ] CHECK constraint em `pipeline.funnel_status` validando: `novo`, `contactado`, `respondeu`, `interessado`, `convertido`, `perdido`
- [ ] CHECK constraint em `sdr_activities.prospect_status` validando: `nao_contatado`, `contatado`, `cliente`, `fora_do_ramo`
- [ ] Index parcial em `sdr_activities.next_action_date WHERE NOT NULL`
- [ ] Index composto em `pipeline(funnel_status, entered_stage_at)`
- [ ] UNIQUE index em `sdr_activities(lead_id)` e `pipeline(lead_id)`

2. Auto-creation Trigger
- [ ] Trigger `auto_create_pipeline_and_sdr()` executado em AFTER INSERT na tabela `leads`
- [ ] Ao inserir lead, registro em `pipeline` criado automaticamente com `funnel_status='novo'`
- [ ] Ao inserir lead, registro em `sdr_activities` criado automaticamente com `prospect_status='nao_contatado'`

3. API de Leads
- [ ] `GET /api/leads` retorna lista de leads com paginação (`?limit=&offset=`)
- [ ] `GET /api/leads/:id` retorna lead individual
- [ ] `POST /api/leads` cria lead (trigger auto-cria pipeline + sdr_activities)
- [ ] `PATCH /api/leads/:id` atualiza dados cadastrais

4. API SDR
- [ ] `GET /api/sdr/queue` retorna fila ordenada: cadência vencida primeiro, depois por score (JOIN leads + sdr_activities)
- [ ] `GET /api/sdr/queue` aceita filtros: `?city=`, `?prospect_status=`, `?cadence=overdue|today|planned`
- [ ] `PATCH /api/sdr/:leadId/action` registra ação: incrementa `contact_count`, atualiza `last_contact_at`, avança `cadence_step`, define próxima `next_action_date`
- [ ] `PATCH /api/sdr/:leadId/notes` adiciona nota ao array JSONB (append, não replace)
- [ ] `GET /api/sdr/stats` retorna: total na fila, contatos feitos hoje, pendentes, vencidos

5. API Pipeline
- [ ] `GET /api/pipeline` retorna leads agrupados por `funnel_status` (JOIN leads + pipeline)
- [ ] `GET /api/pipeline/summary` retorna métricas por stage: quantidade, percentual, taxa de conversão para próximo stage, tempo médio no stage
- [ ] `PATCH /api/pipeline/:leadId/move` valida transição via `funnelTransitions` map antes de mover
- [ ] `PATCH /api/pipeline/:leadId/move` ao mover para "perdido" exige `loss_reason`
- [ ] `PATCH /api/pipeline/:leadId/move` grava entrada em `stage_history` JSONB: `{from, to, moved_by, moved_at}`
- [ ] `GET /api/pipeline/history` retorna movimentações recentes (últimas 50)

6. Quality
- [ ] Testes unitários para leads-service, sdr-service, pipeline-service
- [ ] Testes de integração para todas as rotas
- [ ] `npm run lint` passa
- [ ] `npm run typecheck` passa
- [ ] `npm test` passa

## Dev Notes

### Schema SQL (Architect-Validated)

Usar **exatamente** o SQL do brief `docs/brief-sdr-funil.md` v2.0 seção "Schema Validado pelo @architect". Pontos críticos:

- `lead_id` é `BIGINT NOT NULL` (não INTEGER) — `leads.id` é BIGSERIAL
- `pipeline.funnel_status` é `NOT NULL DEFAULT 'novo'`
- Index parcial: `CREATE INDEX idx_sdr_next_action ON sdr_activities(next_action_date) WHERE next_action_date IS NOT NULL;`
- Index composto: `CREATE INDEX idx_pipeline_status_entered ON pipeline(funnel_status, entered_stage_at);`

```javascript
const funnelTransitions = {
  novo: ['contactado', 'perdido'],
  contactado: ['respondeu', 'perdido'],
  respondeu: ['interessado', 'perdido'],
  interessado: ['convertido', 'perdido'],
  convertido: ['perdido'],
  perdido: ['novo']
};
```

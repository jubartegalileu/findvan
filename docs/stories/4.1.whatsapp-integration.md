# Story 4.1: WhatsApp Integration & Message Queue

**Epic:** Module 4 - WhatsApp SDR Automation
**Story ID:** 4.1
**Status:** Done
**Assigned To:** @dev (Dex)

## Description

Implementar integração com WhatsApp Business API e criar sistema robusto de fila de mensagens com ratelimiting, permitindo ao usuário enviar mensagens automatizadas para leads capturados, com templates customizáveis e rastreamento de entrega/resposta.

**User Story:**
> Como usuário SDR, quero enviar mensagens WhatsApp automatizadas para múltiplos leads, com controle de taxa de envio e rastreamento de status, para que eu possa escalar meu outreach sem manual work.

---

## Problem & Context

**Current State:**
- Dashboard exibe leads (Module 3 ✅)
- Dados de leads armazenados em PostgreSQL (Module 2 ✅)
- Sem forma de contatar leads automaticamente
- Sem tracking de conversas

**Target State:**
- Envio de mensagens WhatsApp via API
- Fila de mensagens com ratelimiting (5 msg/min)
- Templates customizáveis por campaign
- Rastreamento de status (enviado, entregue, lido, respondido)
- Schedule de envios para horários específicos

## Technical Details

### Technology Stack

| Component | Tech | Rationale |
|-----------|------|-----------|
| **Queue** | Bull/BullMQ | Production-ready, Redis or in-memory |
| **WhatsApp API** | Twilio | Free sandbox, good documentation, webhook support |
| **Persistence** | PostgreSQL | Reuse Module 2 DB, ACID transactions |
| **Webhooks** | Express + crypto | Verify Twilio signature, handle retries |
| **Scheduling** | node-cron | Trigger campaigns at specific times |

### Database Schema

**Table: `message_queue`**
```sql
CREATE TABLE message_queue (
  id BIGSERIAL PRIMARY KEY,
  lead_id INTEGER NOT NULL REFERENCES leads(id) ON DELETE CASCADE,
  campaign_id INTEGER NOT NULL REFERENCES campaigns(id) ON DELETE CASCADE,
  phone VARCHAR(20) NOT NULL,
  template_id INTEGER NOT NULL REFERENCES templates(id),
  message_text TEXT NOT NULL,
  status VARCHAR(20) DEFAULT 'pending', -- pending, sent, delivered, failed, read, responded
  twilio_message_id VARCHAR(100),
  attempt_count INTEGER DEFAULT 0,
  next_retry_at TIMESTAMP,
  created_at TIMESTAMP DEFAULT NOW(),
  sent_at TIMESTAMP,
  delivered_at TIMESTAMP,
  failed_reason TEXT,
  metadata JSONB, -- campaign context, variables used, etc
  updated_at TIMESTAMP DEFAULT NOW()
);
CREATE INDEX idx_queue_status_created ON message_queue(status, created_at DESC);
CREATE INDEX idx_queue_lead ON message_queue(lead_id);
CREATE INDEX idx_queue_campaign ON message_queue(campaign_id);

-- Trigger: auto-update updated_at
CREATE TRIGGER update_message_queue_timestamp
  BEFORE UPDATE ON message_queue
  FOR EACH ROW EXECUTE FUNCTION update_timestamp();
```

**Table: `templates`**
```sql
CREATE TABLE templates (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  content TEXT NOT NULL,
  variables JSONB, -- list of {{variable}} names used
  is_default BOOLEAN DEFAULT false,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);
```

**Table: `campaigns`**
```sql
CREATE TABLE campaigns (
  id SERIAL PRIMARY KEY,
  name VARCHAR(150) NOT NULL,
  template_id INTEGER NOT NULL REFERENCES templates(id),
  filters JSONB, -- {city: "São Paulo", source: "google_maps", is_valid: true}
  schedule_config JSONB, -- {type: "immediate"|"scheduled", time: "14:00", daysOfWeek: [1,2,3]}
  rate_limit_per_minute INTEGER DEFAULT 5,
  status VARCHAR(20) DEFAULT 'draft', -- draft, active, paused, completed
  stats JSONB, -- {total: 100, sent: 50, delivered: 45, failed: 5, responded: 10}
  created_at TIMESTAMP DEFAULT NOW(),
  started_at TIMESTAMP,
  completed_at TIMESTAMP,
  updated_at TIMESTAMP DEFAULT NOW()
);
```

### API Endpoints

```
// Campaign Management
POST   /api/campaigns                 - Create campaign
GET    /api/campaigns                 - List campaigns (paginated)
GET    /api/campaigns/:id             - Get campaign details + stats
PATCH  /api/campaigns/:id             - Update campaign (pause/resume)
DELETE /api/campaigns/:id             - Delete campaign (only if draft)

// Templates
GET    /api/templates                 - List all templates
POST   /api/templates                 - Create template
PUT    /api/templates/:id             - Update template
DELETE /api/templates/:id             - Delete template (check usage)

// Message History
GET    /api/messages?lead_id=X        - Get messages for lead
GET    /api/messages?campaign_id=Y    - Get campaign messages
GET    /api/messages/:id              - Get single message details

// Webhook (Twilio → Backend)
POST   /api/webhooks/twilio           - Webhook receiver (with signature verification)

// Queue Status (admin/monitoring)
GET    /api/queue/stats               - Queue size, throughput, avg latency
POST   /api/queue/retry/:message_id   - Manually retry failed message
```

## Dev Notes

**Decisions Made:**
1. **Twilio vs WhatsApp Direct API:** Twilio chosen for simplicity + webhook support
2. **Queue: Bull/BullMQ vs AWS SQS:** Bull chosen for local dev + free tier
3. **Persistence:** PostgreSQL + in-memory for flexibility
4. **Rate Limiting:** Token bucket algorithm (industry standard)
5. **Webhooks:** Sync + signature verification (Twilio requires it)

**Known Risks:**
- Twilio sandbox session expires 24h (need refresh during testing)
- Rate limits may affect production (monitor and adjust)
- Phone validation complex for international numbers

**Testing Strategy:**
- Mock all Twilio calls (don't spend API quota in tests)
- Test queue with in-memory Bull instance
- Test webhooks with mock signatures
- Integration test: DB → Queue → Mocked Twilio → DB update

## Story Summary

Adicionar ciclo de vida ao lead com 6 estagios de funil (Novo, Contactado, Respondeu, Interessado, Convertido, Perdido), permitindo ao SDR acompanhar a jornada de cada lead desde a captacao ate a conversao ou perda.

**Goal:** Todo lead possui um status de funil visivel, editavel via dropdown, com historico de transicoes e motivo obrigatorio para perdas.

---

## Story Description

### Problem/Need
Atualmente todos os leads tem um unico status "Qualificado" — nao ha como saber se um lead ja foi contactado, se respondeu, ou se virou cliente. O SDR abre a tela de Leads e nao sabe onde focar esforco. Leads quentes (que responderam) parecem iguais a leads frios (nunca contactados).

### Solution
Implementar campo `funnel_status` com 6 estagios e transicoes definidas. Incluir motivo obrigatorio ao marcar como "Perdido". Registrar historico de mudancas na tabela `lead_interactions` para audit trail. UI com badge colorido por status e dropdown para transicao.

### User Value
"Como SDR, quero ver em qual estagio do funil cada lead esta para focar meu esforco nos leads quentes e nao recontactar quem ja foi abordado."

   - [x] 6 estagios implementados como enum:
     - `novo` — Lead recem-importado (default)

     - `respondeu` — Lead respondeu mensagem
     - `interessado` — SDR marcou como interessado
     - `convertido` — Lead virou cliente
     - `perdido` — SDR descartou o lead
   - [x] Todo lead novo recebe status "novo" automaticamente

2. **Transicoes Permitidas**
   - [x] Novo → Contactado (manual ou automatico ao enviar msg)
   - [x] Contactado → Respondeu (automatico ao receber resposta)
   - [x] Respondeu → Interessado (manual pelo SDR)
   - [x] Interessado → Convertido (manual pelo SDR)
   - [x] Qualquer estagio → Perdido (manual pelo SDR)

## Story Tasks

### Task 1: Backend — Schema e Migration
- [x] Migration Alembic: adicionar campo `funnel_status VARCHAR(20) DEFAULT 'novo'` em leads
- [x] Migration Alembic: adicionar campo `loss_reason VARCHAR(50)` em leads (nullable)
- [x] Migration Alembic: criar tabela `lead_interactions`:
  ```sql
  CREATE TABLE lead_interactions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    lead_id UUID REFERENCES leads(id) ON DELETE CASCADE,
    type VARCHAR(30) NOT NULL,
    content TEXT,
    metadata JSONB,
    author VARCHAR(100),
    created_at TIMESTAMP DEFAULT NOW()
  );
  CREATE INDEX idx_interactions_lead_id ON lead_interactions(lead_id);
  CREATE INDEX idx_interactions_created_at ON lead_interactions(created_at);
  ```

## Riscos

| Risco | Impacto | Mitigacao |
|-------|---------|-----------|
| Transicoes rigidas podem frustrar usuario em cenarios reais | Medio | Permitir override com confirmacao ("Transicao incomum — confirmar?") se necessario no futuro |
| Sub-formulario de motivo pode ser friction no fluxo rapido | Baixo | Motivo mais comum pre-selecionado; campo Outro so aparece se selecionado |
| Tabela lead_interactions pode crescer rapido | Baixo | Indice em lead_id + created_at ja planejado; cleanup de registros > 1 ano no futuro |
| Complexidade maior que estimada (maquina de estados + nova tabela) | Medio | Elevado para 5 pontos; se estender, dividir Task 5 (Dropdown) em story separada |

# Architect Gate — Story 15.1 (Dedicated Worker & Distributed Lock for Retention)

**Date:** 2026-02-28  
**Agent:** @architect  
**Status:** PASS

## Scope Reviewed

- Isolamento do retention runtime em worker dedicado, removendo loop periódico do processo FastAPI.
- Coordenação multi-instância com lock distribuído (TTL + heartbeat + failover).
- Preservação de contrato dos endpoints de status e resiliência operacional.

## Architectural Validation

1. Worker isolation
- `retention_jobs_service` foi refatorado para operação orientada a worker (`run_retention_worker`) sem thread daemon local no processo da API.
- `main.py` mantém inicialização de estado/snapshot via `start_retention_job`, sem iniciar ciclo de execução periódico interno.
- Novo entrypoint dedicado `app/cli/retention_worker.py` habilita execução separada do job.

2. Distributed lock and failover
- Novo `distributed_lock_service` implementa aquisição/renovação/liberação de lock com TTL em tabela compartilhada (`distributed_job_locks`).
- Renovação por heartbeat evita perda de ownership durante ciclo; expiração do lock permite takeover seguro por outra instância.
- Cenário de lock indisponível mantém API estável e sem impacto nos endpoints principais.

3. Shared status and compatibility
- Estado operacional persistido em `retention_job_status` para visibilidade cross-process (running, owner, last_run, last_success, métricas de execução).
- Endpoints existentes que retornam `get_retention_job_status` mantêm payload backward compatible.
- Fallback em memória preserva degradação graciosa quando banco não estiver disponível.

4. Validation evidence
- Testes adicionados para lock distribuído (`test_distributed_lock_service.py`) e comportamento do worker (`test_retention_jobs_service.py`).
- Quality gates do repositório executados com sucesso: `npm run lint`, `npm run typecheck`, `npm test`.

## Residual Risks (non-blocking)

- Fallback em memória para lock/status é útil para dev/test, mas em produção o modo recomendado é com PostgreSQL disponível para garantir coordenação real entre instâncias.
- Recomenda-se operação do worker dedicado sob supervisor (systemd/PM2/K8s) com política de restart.

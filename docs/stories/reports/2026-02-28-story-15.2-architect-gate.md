# Architect Gate — Story 15.2 (Persistent Operational State Store)

**Date:** 2026-02-28  
**Agent:** @architect  
**Status:** PASS

## Scope Reviewed

- Persistência de estado operacional de retention/alerting em storage compartilhado.
- Compatibilidade de contrato para endpoints já consumidos pela UI.
- Consistência de leitura/escrita em cenário multi-instância com fallback controlado.

## Architectural Validation

1. Schema e persistência
- `db.py` adiciona tabelas `alerting_state` e `alerting_recent_events` para estado e histórico recente de alertas.
- Estado de retenção já persistente em `retention_job_status` (entregue na Story 15.1) permanece compatível com o objetivo da Story 15.2.

2. Migração com fallback
- `alerts_service` foi migrado para leitura/escrita persistente com sincronização de memória via `_sync_state_to_memory`.
- Em indisponibilidade de banco, serviço mantém operação com fallback em memória sem quebrar fluxo.

3. Consistência e contrato
- Leituras de status (`get_alerting_status`) priorizam estado persistente, garantindo consistência cross-instance.
- Escritas de contadores, cooldown e eventos recentes são persistidas com upsert/append e trimming controlado.
- Contrato da API de alerting permanece backward compatible (campos existentes preservados).

4. Validation evidence
- Quality gates verdes no ciclo: `npm run lint`, `npm run typecheck`, `npm test`.
- Suite de backend passou sem regressão funcional.

# Architect Gate — Story 16.1 (Automated Recovery Policies for Retention/Alerting)

**Date:** 2026-02-28  
**Agent:** @architect  
**Status:** PASS

## Scope Reviewed

- Políticas de self-healing para retenção e alerting com budget/cooldown/backoff.
- Mecanismo de circuit-breaker para evitar loops agressivos de remediação.
- Compatibilidade dos contratos existentes com exposição aditiva de status de recuperação.

## Architectural Validation

1. Recovery safety model
- `retention_jobs_service` implementa estado de recuperação dedicado com:
  - `window_seconds` + `max_attempts` (budget por janela)
  - `cooldown_seconds` + `backoff_seconds`
  - `circuit_breaker_threshold` + `circuit_breaker_seconds`
- Fluxo de recuperação é acionado somente após falha de ciclo, com execução controlada e bloqueio por circuit-breaker quando necessário.

2. Alerting self-healing
- `alerts_service` aplica política equivalente para tentativas de reenvio pós-fallback.
- Recuperação assíncrona evita bloqueio do request path.
- Fallback/retry/suppress permanecem isolados e não derrubam API.

3. Observability and traceability
- Ações de remediação são registradas em `operational_incidents` com causa, tentativa, sucesso/falha e motivo de skip.
- `get_alerting_status` e `get_retention_job_status` expõem snapshot de self-healing (payload aditivo).
- Endpoints aditivos (`/api/dashboard/recovery-policies` e telemetria operacional) preservam backward compatibility.

4. Validation evidence
- Testes cobrindo comportamento de recuperação adicionados/atualizados:
  - `test_retention_jobs_service.py`
  - `test_alerts_service.py`
  - `test_dashboard_api.py`
- Quality gates verdes no ciclo: `npm run lint`, `npm run typecheck`, `npm test`.

## Residual Risks (non-blocking)

- Em produção, recomenda-se ajuste fino de limites de recuperação por ambiente para evitar excesso de tentativas sob incidentes prolongados.
- Monitorar taxa de incidentes `*_recovery_skipped` para calibrar budget/circuit-breaker.

## Architectural Validation

1. Incident classification model
- `operational_telemetry_service` introduz classificador com versão (`INCIDENT_CLASSIFIER_VERSION = 1.0.0`).
- Incidentes recebem enriquecimento aditivo (`classification`) com:
  - severidade normalizada (`low/medium/high/critical`)
  - componente afetado (`retention/alerting/messaging`)
  - impacto operacional
  - recomendação acionável
- Modelo é deterministicamente derivado de `source/event_type` e auditável por versão.

2. Guardrails contract
- Endpoint aditivo `GET /api/dashboard/guardrails` entregue com `limit/offset` e `applied_limit/applied_offset`.
- Guardrails combinam sinais de threshold + incidentes classificados e priorizam por severidade.
- `GET /api/dashboard/operational-telemetry` evoluiu de forma compatível (`applied_offset`, `classifier_version`).

3. API/UI consistency
- Dashboard consome guardrails e renderiza recomendações com prioridade/severidade consistente ao backend.
- Não houve remoção de campos existentes nos payloads previamente consumidos.

4. Validation evidence
- Testes de API atualizados (`test_dashboard_api.py`) cobrindo endpoint de guardrails e paginação.
- Quality gates verdes no ciclo: `npm run lint`, `npm run typecheck`, `npm test`.

## Residual Risks (non-blocking)

- Regras de classificação são heurísticas iniciais; recomenda-se revisão periódica dos mapeamentos por incidentes reais.
- Em volume alto de incidentes, ranking pode requerer ajuste fino por contexto de negócio (não só severidade).

# 2026-03-02 — Story 25.1 Backend Audit (SDR/Pipeline)

## Context

Audit inicial pós Epic 24 para priorizar hardening de backend em SDR/Pipeline.

Arquivos-base revisados:
- `packages/backend/app/db.py`
- `packages/backend/app/services/sdr_service.py`
- `packages/backend/app/services/pipeline_service.py`
- `packages/backend/app/api/sdr.py`
- `packages/backend/app/api/pipeline.py`
- `packages/backend/tests/api/test_sdr_api.py`
- `packages/backend/tests/api/test_pipeline_api.py`

## Findings (prioridade)

1. **P0 — Guard rails de volume em trigger/backfill**
- O trigger `auto_create_pipeline_and_sdr()` está correto funcionalmente, mas imports massivos podem pressionar I/O por `FOR EACH ROW`.
- Ação: documentar e testar baseline de import em lote + validar estratégia de execução transacional.

2. **P0 — Performance em queries críticas**
- `/api/sdr/queue` e `/api/pipeline` fazem `JOIN` com ordenações frequentes.
- Índices atuais existem, porém falta validação por cenário de carga real (sem apenas confiar em leitura estática).
- Ação: criar testes/benchmarks mínimos de regressão de tempo para 5k+ leads.

3. **P1 — Padronização de erro e observabilidade**
- Handlers já retornam 400/500, mas faltam sinais operacionais consistentes para troubleshooting (ex.: erro de transição + contexto resumido).
- Ação: normalizar payload de erro em endpoints SDR/Pipeline e adicionar logs estruturados de falhas críticas.

4. **P1 — Cobertura de integração para casos de volume**
- Testes atuais cobrem happy path/validation, mas não pressão de volume.
- Ação: expandir suíte de integração para cenários de alto volume e caminhos de fallback.

## Proposed Execution Plan (Story 25.1)

1. Definir baseline de carga (dataset alvo e tempos esperados).
2. Escrever testes de regressão de performance/consistência (integração).
3. Ajustar índices/queries conforme evidência dos testes.
4. Padronizar erros/logs e validar compatibilidade de contrato.
5. Fechar com relatório de benchmark antes/depois no story.

## Exit Criteria (25.1)

- Sem regressão de contrato em `/api/sdr/*` e `/api/pipeline/*`
- Testes de integração e volume verdes
- Evidência de melhoria/estabilidade documentada
